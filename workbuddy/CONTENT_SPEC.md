# WorkBuddy 专题内容规范

## 专栏目的

记录 WorkBuddy 在真实工作中的使用方法。读者应能带走一个可执行动作，而不是只知道某项产品功能。

专题文章按四层组织：

1. 为什么要把任务交代清楚。
2. WorkBuddy 在一项任务中承担什么角色。
3. 如何完成并验收第一次低风险任务。
4. 哪些任务值得变成团队工作流。

## 每篇文章必须包含

- 一个具体场景或问题作为开头。
- 读者、适用边界和一个明确行动。
- 使用前的材料条件，尤其是授权和数据边界。
- 人工验收动作；不能把专业判断、审批和责任交给工具。
- 资料来源。产品事实必须链接 WorkBuddy 官方文档；本站的方法判断要明确标为实践建议。

## 深度与实操门槛

- 教程型文章必须完整跑通一个任务，不能只介绍观点或功能。
- 至少交付：输入材料、任务设计、两份中间产物、一个失败或错误样例、人工验收记录、最终可复用模板。
- 所需输入、执行步骤、目标、验收标准和沉淀位置必须能被读者直接找到。
- 正文长度由任务复杂度决定；低于 3000 个可见中文字符的内容原则上作为短笔记，不包装成长篇教程。
- 不写未经实际测算的阅读时长。若标注字数或时长，必须由发布脚本或人工统计得到。
- 教学模拟材料必须明确标注，不能写成真实客户案例或亲身效果。

## 证据等级与发布命名

| 等级 | 已有证据 | 可以使用的名称 |
| --- | --- | --- |
| E0 观点 | 只有判断、资料摘录或选题设想 | 观察、短笔记、选题线索 |
| E1 教学模拟 | 有脱敏输入、任务脚本和人工构造的标准输出 | 教学实验、任务脚本、课堂样例 |
| E2 单次实跑 | 在 WorkBuddy 中实际运行，保留输入、过程截图、原始输出和人工验收 | 实操教程、实跑记录 |
| E3 重复验证 | 同一任务至少重复运行，记录失败、修正和版本差异 | 可复现案例、稳定工作流 |
| E4 业务验证 | 来自获准公开的脱敏业务现场，有使用者、验收和后续复盘 | 企业案例、客户案例 |

- 没有达到 E2，不得使用“完整实操”“实测”“跑通”或“效果证明”。
- 没有达到 E4，不得使用“企业成功案例”“客户效果”或暗示真实业务采用。
- 发布页必须显示证据等级、验证日期和仍缺少的证据。

## Case 交付包

深度教程不能只有网页正文。每个 Case 在进入写作前，至少准备：

- `input/`：可公开或脱敏的原始输入，以及字段说明。
- `run/`：任务说明、每轮提示词、WorkBuddy 过程截图和运行时间。
- `output/raw/`：WorkBuddy 原始输出，不做事后美化。
- `output/reviewed/`：人工修订后的交付物和修改说明。
- `acceptance.md`：验收项、结果、未通过项和责任人。
- `lessons.md`：失败、限制、适用条件和下一版动作。
- `assets/`：截图、GIF、视频及其来源、脱敏和授权说明。

模板见 `workbuddy/CASE_TEMPLATE.md`。没有输入包、原始输出或验收记录的页面，不进入实操教程栏目。

## 事实与方法的边界

- 官方页面可用于说明产品界面和公开功能，但不复制大段原文。
- 没有亲自验证的效果、效率、客户案例和数字不写。
- 产品能力、权限、连接器和价格会变化；每次更新前核对官方页面，并在文章中写明适用条件。
- 遇到涉密、客户、个人信息或受控资料，先说明企业允许的环境与授权要求，不提供绕过权限的指引。

## 图片与发布

- 图片必须承担信息任务，例如证明输入、执行过程、中间产物、人工修改或沉淀位置。
- 每篇深度教程至少有三处信息型视觉；官方产品截图不能作为主要视觉，也不能代替业务产物。
- E2 以上案例至少包含一张真实输入截图、一张 WorkBuddy 执行过程截图和一张结果或交付物截图。CSS 信息图只能辅助解释，不能充当运行证据。
- 截图应紧跟对应步骤，说明读者此处要看什么；需要核对字段时应增加标注或局部放大。
- 产品截图仅用于解释具体操作；在图片说明和 `assets/SOURCES.md` 中保留来源。
- 新文章要同时更新 `workbuddy/index.html` 的卡片和首页 `index.html` 的最新文章入口。
- 发布前检查桌面和手机宽度、全部本地链接、图片加载、文字是否有空泛 AI 腔。
- 推送到 `main` 后，访问生产地址确认页面和静态素材可用。
